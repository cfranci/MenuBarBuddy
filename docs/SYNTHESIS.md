# Cross-Section Synthesis (Armada, 2026-07-08)

Five sections cohere into one launch plan for MenuBarBuddy, and the through-line is consistent across all of them: ship DIRECT ONLY (no Mac App Store for v1) as a Developer-ID-signed, notarized, stapled DMG on a Lemon Squeezy checkout at exactly $4.20, with Sparkle 2 auto-updates from a self-hosted EdDSA appcast, and the GitHub repo kept PUBLIC but relicensed source-available. The distribution and licensing-payments sections independently reach the SAME core technical conclusions and reinforce each other cleanly: (1) install.sh's ad-hoc codesign (verified: `codesign --force -s -` at install.sh:12) is the #1 ship blocker and must become Developer-ID + hardened-runtime + notarize + staple; (2) there is NO LICENSE file (verified), so 'open source' is not yet legally true and a license must be committed; (3) the self-repair watchdog (verified: performSelfRepair rewrites CFBundleIdentifier and shells out `codesign --force -s -; open` to re-sign ad-hoc at runtime) is both the product's moat AND the reason MAS is off the table, since runtime re-signing, bundle-ID mutation, and CGWindowList inspection are all sandbox-forbidden. licensing-payments adds the critical, code-grounded refinement that the license token must ride the existing UserDefaults bundle-ID migration chain (SettingsStore.migrateFromOldBundleID) rather than Keychain, or a watchdog-triggered self-repair could strand paying customers. brand-copy (keep the MenuBarBuddy name, benefit-first voice, $4.20 joke landed once per surface, Bartender-trust-scandal positioning) and the two production sections (landing-page site at /Users/cf/Projects/MenuBarBuddy/site/index.html, and the assets capture runbook) are all mutually consistent EXCEPT for one significant factual conflict about what the app's toggle icon actually is (dot vs folder), plus two smaller asset/feature mismatches, detailed in conflicts. Net: the strategy is sound and internally reinforced; the remaining work is (a) fix the dot-vs-folder error in the landing page + README, (b) reconcile the custom-icons and asset-filename mismatches, and (c) resolve the launch-blocking unknowns (does $4.20 exist on MAS — moot for v1; does self-repair break the notarized build's relaunch under Gatekeeper — the mandatory fresh-Mac acceptance gate; which license to commit).

## CONFLICTS
- DOT vs FOLDER — the single most important conflict, and the landing-page section got it backwards. The landing-page section deliberately rebuilt the whole demo around a FOLDER glyph (📁) as the default, citing README.md line 9 and treating the assets section as wrong. But the running code renders a DOT: AppDelegate.swift:537 sets `button.image = makeDotImage(color: grayColor)` and line 539 sets `button.title = ""` (blanking the emoji), and the rainbow path at line 383 uses `makeWaveDotImage(...)` — also a dot. The `📁` at SettingsStore.swift:25 is only a stored default that the runtime button never displays (the emoji picker is the known dead no-op flagged in the pre-launch review). So the assets section is RIGHT (ship shows a dot), the brand-copy section is RIGHT (its bullets say 'dot' / 'Rainbow Dot'), and the landing-page section is WRONG — its folder rebuild is grounded in stale README language, the exact defect it claimed to be fixing. ACTION: the landing-page copy and demo must be re-flipped from folder back to dot before ship. README.md is also stale (lines 9-32 say 'Folder 📁') and should be corrected so this does not bite again.
- ASSET FILENAME: icons.png vs menu.png — the assets runbook (Section 5, 'REQUIRED by index.html') lists `icons.png` as a required asset and scripts capturing the Choose Icons panel to `assets/icons.png`. But the on-disk site/index.html actually references `assets/menu.png` (verified by grep), NOT icons.png, and the landing-page section explicitly repointed that third screenshot away from the broken Choose Icons panel to the right-click menu (menu.png) because the icon picker is a dead no-op. So the assets runbook is capturing the wrong screenshot to the wrong filename. ACTION: capture the right-click context menu and save it as `assets/menu.png` (as the landing-page section intends); do NOT produce `assets/icons.png` (nothing references it) and do NOT capture the Choose Icons panel as a hero asset.
- CUSTOM ICONS as a selling point — brand-copy lists 'Custom icons: Yes' in its comparison table and 'Make it yours. Pick any emoji or symbol' as a feature bullet and in the App Store description. The landing-page section (correctly, per the pre-launch review) stripped all custom-icon claims because the emoji picker is a verified dead no-op and the start-marker emoji reverts on next expand. These two sections make opposite claims about a feature that does not work. ACTION: adopt the landing-page section's position everywhere — remove custom-icon claims from brand-copy's comparison table, feature bullets, and store description; scope customization to what verifiably works (classic dot vs Rainbow Dot).
- TRIAL LENGTH: licensing-payments specifies a 14-day degrade-never-brick trial; brand-copy's FAQ says there is no packaged trial yet ('a packaged trial and license flow are planned, not yet shipped') and frames 'build it from source' as the de-facto trial. These describe different states (built vs planned). Not a hard contradiction, but the public-facing copy and the implementation plan must agree at ship time. ACTION: if the 14-day trial ships, update brand-copy's 'Is there a free trial?' FAQ to describe it; if it does not ship in v1, licensing-payments' trial is roadmap, not launch.
- SPARKLE INITIALIZER / NAMESPACE fabrications — both distribution and licensing-payments independently warn that source drafts fabricated a bogus Sparkle XML namespace (andymatriculia / andymatuschak misspellings) and a non-existent `startUpdater(true)` API. They agree on the correction (real namespace `http://www.andymatuschak.org/xml-namespaces/sparkle`, real initializer `SPUStandardUpdaterController(startingUpdater:updaterDelegate:userDriverDelegate:)`). No conflict between the two, but flagging because the wrong values appear in sibling drafts and must not be copied into the actual Info.plist/appcast.
- LICENSE STORAGE: Keychain vs UserDefaults — not a cross-section conflict (only licensing-payments covers it) but the highest-severity single-section claim, so surfacing it. licensing-payments argues the license token MUST live in UserDefaults on the bundle-ID migration chain (SettingsStore.migrateFromOldBundleID), NOT Keychain, because self-repair bumps the bundle ID and ad-hoc re-signs, which can strand a Keychain item. This is verified against the code (performSelfRepair rewrites CFBundleIdentifier via SettingsStore.nextBundleID then re-signs). Any future contributor's instinct to 'just use Keychain for secrets' would silently break paying customers on the next watchdog event. Must be honored.

## OPEN_QUESTIONS
- Is exactly $4.20 selectable in App Store Connect for the US storefront? The entire brand (the meme price) and part of the distribution argument hinge on this, yet distribution itself flags it as 'the shakiest assertion in the whole analysis.' It does not block the direct-sale decision (the sandbox incompatibility with self-repair does that independently), but it should be verified before any MAS v2 is contemplated, and before brand-copy hard-commits to '$4.20 everywhere, never discount.'
- Does self-repair actually need to fire on a notarized, Developer-ID-signed build in the field — and if so, does the Gatekeeper-blocked-relaunch bug reproduce on a stock Mac? Distribution calls this the ship-blocking bug and prescribes a fix (flag-gate the runtime bundle-ID-mutation + ad-hoc re-sign behind `#if AD_HOC_BUILD`, keep detection, degrade to Repair-and-Relaunch or fresh-download-and-replace). This is unproven until someone force-triggers `--force-repair` on a Mac that never built the app. It is the mandatory acceptance gate in both distribution and licensing-payments and the biggest technical unknown for launch.
- What is the license actually going to be? distribution recommends adding PolyForm Noncommercial 1.0.0 and stresses there is NO LICENSE file today (verified: no LICENSE in repo), meaning the code is currently all-rights-reserved and 'open source' is not yet legally true. brand-copy and landing-page both lean on an 'open source, build it yourself' trust pitch. Until the LICENSE is chosen and committed, that pitch is not accurate. Chase needs to pick: PolyForm NC (source-available, blocks fork-and-resell) vs a permissive OSS license vs staying source-visible-only. This blocks the honesty of the marketing.
- Does the '3 seat / 3 activation' cap (licensing-payments) reconcile with distribution's framing? distribution treats DRM as theater at $4.20 and says not to invest in enforcement; licensing-payments builds a real Ed25519-signed token + server-side 3-activation cap via a Cloudflare Worker. These are compatible (light enforcement) but the seat count and whether the Worker is even worth standing up for a $4.20 app is a product call worth confirming.
- Custom domain menubarbuddy.app — distribution mentions it as optional (~$15/yr) and the site uses a {SITE_URL} placeholder for canonical/OG; landing-page and assets both leave {BUY_URL}, {SUPPORT_EMAIL}, {SITE_URL} as unresolved placeholders. Someone must decide the domain and fill all placeholders before the page and OG tags are correct.
- Will the 'before' screenshot look cluttered enough? The assets runbook honestly notes the hidden zone is mostly transient system indicators and a thin 'before' shot is real system state, not a bug — but the entire brand narrative ('your menu bar was hiding 12 icons') depends on a visibly full before-state. Open question whether to stage extra menu-bar apps before capture to make the before/after contrast land.

# MenuBarBuddy Pre-Launch Review

This is the distilled result of a six-section ensemble code review (front-end, back-end, design, user-experience, security, performance, plus a reconciling synthesis) run before commercializing MenuBarBuddy at $4.20. The sections converged independently on the same handful of defects, which raises confidence. Items that would embarrass a paid app are flagged under "Fix before charging money"; everything is deduplicated across sections, every file:line below was re-verified against the current source, and nothing in this doc proposes touching the hardened Tahoe collapse/watchdog/self-repair machinery (see HANDOFF.md), except where a fix is explicitly scoped to be length-safe. All paths are relative to `Sources/MenuBarBuddy/`.

---

## Fix before charging money

These are the bugs a paying customer would hit in the first session or the first week.

### Dead and lying Settings UI

- **The top "Folder Icon (toggle)" picker does nothing, ever.** The window's largest, top-most control (GroupBox at `Views/SettingsView.swift:21-50`, plus half the quick-pick grid) calls `selectEnd` -> `onEmojiChange` -> `toggleItem.button?.title = emoji` (`AppDelegate.swift:868`). But the toggle button is image-only with an empty title (`AppDelegate.swift:538-539`), and its image is repainted every animation frame (`:420` color fade, `:383` rainbow), so the title is never drawn. `store.selectedEmoji` persists but is never read back at launch. A paid app whose headline setting silently no-ops reads as broken on day one. Fix: this is a product decision (see Open Questions). All three UI sections lean the same way, delete the custom toggle emoji feature and replace the section with a working Dot Style control (Green / Rainbow, bound to the existing `MenuBarBuddy.rainbowDot` flag) with a live preview from the real `makeDotImage`/`makeWaveDotImage`.
- **The two customizations that actually work are hidden.** Rainbow Dot (`AppDelegate.swift:584`) and Start at Login (`:578`) live only in the right-click context menu of a 9px dot, a gesture new users never attempt. The Settings window omits the settings that work and foregrounds the ones that don't. Fix: add both as real Toggles in SettingsView, refactor `toggleRainbowDot`/`toggleStartAtLogin` to accept a desired state (not a blind `.toggle()`) so window and menu stay in sync, keep the menu entries.
- **Start at Login can silently lie.** `toggleStartAtLogin` swallows SMAppService errors to NSLog only (`AppDelegate.swift:194-196`), so the checkmark can flip even when registration fails; the customer finds out at next reboot. Fix: after toggling, read back `startAtLoginEnabled` (`:173-175`) and alert on disagreement ("try moving MenuBarBuddy to /Applications").
- **The start-marker emoji never persists.** `selectStart` writes the separator title (`AppDelegate.swift:872-874`), but `expandMenuBar()` unconditionally rewrites it back to the hardcoded "│" (`:759-766`) and `configureSeparatorButton()` hardcodes "│" at launch (`:547`). A picked marker is wiped on the next expand. A setting that visibly reverts itself is a refund generator. Fix: make both sites read `store.startEmoji`, preserving the existing font/color/paragraph attributes (this is length-safe, glyph and color only, the length path must not change), or delete the Start Marker section (Open Questions). Either way, dedupe the two identical NSAttributedString constructions (`:551-558` vs `:759-766`) into one shared property.
- **"Still seeing items? Push harder" is a mystery button that becomes a literal no-op.** `SettingsView.swift:119` does `store.pushMultiplier *= 2` with no ceiling shown, no current value, no feedback. The real cap is `min(10_000, max(500, width * 2.5 * mult))` at `AppDelegate.swift:244`, so past roughly 2.3x on a 1728px screen the button does nothing while appearing to work, and the unbounded multiplier persists forever (`SettingsStore.swift:28` clamps only the floor). Fix: replace with a bounded, outcome-framed control (design proposed a 1-4 stepper/slider under an Advanced DisclosureGroup showing the current value); auto re-run the collapse after a change so the effect is immediate; clamp `pushMultiplier` to 1...8 at the model layer regardless of UI (security Finding 7).

### Leaks and races a multi-day daemon will hit

- **Settings window leaks an observer and a retained window on every open.** `openSettings()` registers a `willCloseNotification` block observer and never removes it (`AppDelegate.swift:900-907`); with `isReleasedWhenClosed = false` (`:892`) each open/close cycle stacks another observer and leaks another window. Unbounded over a session that runs for days. Fix: capture the observer token and remove it inside the close block, with `[weak self]`.
- **The safety watchdogs go blind during exactly the drags they guard.** All three timers use `Timer.scheduledTimer` (`AppDelegate.swift:352`, `:406`, `:435`), which registers in the `.default` run-loop mode only. The layout watchdog (~3s) and hosting watchdog (~5s) piggyback on `mouseTimer` (`:453-460`), so during menu tracking and window drags, the moments most likely to misorder or orphan the items, none of them fire. This quietly defeats the hardening a paid customer is relying on. Fix: construct the timers manually and `RunLoop.main.add(t, forMode: .common)`; add `.tolerance` to `mouseTimer` and `rainbowTimer` in the same edit (free energy win).
- **The `safelySetLength` fallback ladder is inverted.** `AppDelegate.swift:713` filters `{ $0 <= target }`, so the expand path (target 12) gets exactly one attempt with no fallback. The fix must be direction-aware, not just deleting the filter (an unfiltered ladder would try 5000 on expand and collapse the bar): use `[target] + [5000, 2500, 1200].filter { $0 < target }`.
- **No `applicationWillTerminate`; teardown is partial.** `quitApp()` (`AppDelegate.swift:910-916`) invalidates only `mouseTimer`, leaves `rainbowTimer` and `colorAnimTimer` live, never cancels the SIGTERM/SIGINT dispatch sources, and calls `NSApp.terminate` immediately after `expandMenuBar()` with no run-loop turn for the host to commit the change. Dying mid-manipulation is the documented Tahoe blacklist trigger, so this is safety, not hygiene. Fix: one `invalidateRuntime()` helper called from both `applicationWillTerminate` and `quitApp`, plus a short settle (`RunLoop.main.run(until:)` ~0.25s) after the expand.

### Security of the self-repair path (small diffs, close them before selling)

- **Shell interpolation in `performSelfRepair()`.** `AppDelegate.swift:848-851` builds a `/bin/sh -c` string with `bundlePath` interpolated. Latent, not live (the path is `/Applications/...` today), but it becomes an injection primitive the moment the app ever runs from an attacker-influenceable path, and the result gets ad-hoc re-signed and relaunched. Fix: pass the path as a positional argument (`arguments: ["-c", script, "_", pid, bundlePath]` with `"$1"`/`"$2"` in the script) so no metacharacter is ever parsed. Smallest diff in the review, closes the only injection primitive in the app.
- **Self-repair is a signing oracle.** `performSelfRepair()` (`AppDelegate.swift:831-853`) runs `codesign --force -s -` over the live in-place bundle with no integrity check, so any on-disk patch to the binary gets re-blessed and relaunched from /Applications. Fix ladder: verify before re-signing (`codesign --verify --deep --strict`, abort repair on failure), re-sign a temp copy rather than the live bundle (gated on the Tahoe blacklist open question below), and adopt hardened runtime (`--options runtime`) everywhere the app is signed, including `install.sh`. Also make the plist rewrite atomic (`.atomic`) and check codesign/open exit status before terminating (`:841-842`, `:852-853`).

## Worth doing soon

- **First-run onboarding.** The whole product hinges on one non-obvious rule (icons left of the boundary hide; Cmd-drag to arrange) that lives only in the README and a confusing 4-word caption (`SettingsView.swift:76`). UX rates a one-time popover with a labeled diagram and a live "try it" collapse as the retention make-or-break for a paid app. Gate on a persisted UserDefaults flag.
- **Make silent failures visible.** Collapse can no-op or auto-revert (`AppDelegate.swift:737-742`, `:746-751`) and `healLayout()` visibly yanks the dot (`:697-705`) with only NSLog. Add a transient tooltip/HUD so safe-but-invisible behavior stops reading as glitches.
- **Collapse post-verify race.** The 1.5s verification (`AppDelegate.swift:746-751`) has no attempt identity and captures `[self]` strongly; a fast collapse/expand/collapse can be spuriously reverted. Add a per-attempt UUID guard cleared in `expandMenuBar`, plus `[weak self]`.
- **Debug harnesses ship in the release binary.** Six unguarded argv flags (`AppDelegate.swift:77-142`); `--force-repair` (`:77`) fires the self-modifying re-sign path with no confirmation, bypassing `showRepairAlert` (`:812`), and the `--test-collapse` path's `dumpStatusWindows` (`:202-212`) enumerates other apps' menu-bar windows via CGWindowList. Recommended: `#if DEBUG` them and confirmation-gate `--force-repair` even in DEBUG, but HANDOFF.md documents them as intentionally kept, so this needs Ga'noh's call (Open Questions).
- **Boundary "│" is nearly invisible.** 8pt ultraLight white at 30% alpha (`AppDelegate.swift:555`) vanishes on a light menu bar, making the core Cmd-drag interaction unteachable. Switch to a template image or `tertiaryLabelColor`. Color/rendering only: the separator's length is load-bearing for collapse (`:520`, `:737`, `:756`), the glyph swap must not touch how length is set.
- **Settings window structural cleanup.** 350x540 hardcoded in three places (`SettingsView.swift:140`, `AppDelegate.swift:881`, `:884`); dead trailing Spacer (`SettingsView.swift:137`); redundant "MenuBarBuddy" title (`:17`); quick-pick grid clips 6 of its 9 rows at a fixed 140pt height (`:113`) with no scroll affordance. Drive size from content; required once the new toggles land or the fixed height will clip them.
- **Quick-pick grid ambiguity.** With `focusedField == nil` a tap silently falls to `selectEnd` (`SettingsView.swift:94-98`) and both roles highlight with one color (`:104`). Use an explicit activeSlot state, an "Editing: Dot / Marker" indicator, and two-color highlights. If any text-entry path survives the product decision, replace `newValue.last` (`:145-156`) with `String(newValue.suffix(1))` so multi-scalar emoji (flags, ZWJ, keycaps) don't split; design's stronger move is deleting the TextField and making the swatch the button (keep the Character Viewer as "More...").
- **Multi-display correctness.** `updateCollapseLength()` (`AppDelegate.swift:243`) and `pollMouseLocation()` (`:442`, magic `- 25` at `:446`) key off `NSScreen.main` instead of the item's own screen; collapse can be incomplete and hover can fail on a wider secondary display. Promote to fix-before-charging if Ga'noh (or customers) run multi-display, which for a paid audience is likely.
- **Rainbow/energy polish.** Idle-gate the 20Hz per-pixel rebuild when collapsed + idle + alpha 0 (`AppDelegate.swift:362-383`), but keep advancing both `rainbowHue` and `rainbowWavePhase` or the wave pops on re-expand; reuse the `NSBitmapImageRep` (`:295`); hoist `sin()` per-column out of the inner loop (`:311-335`, `:323`); cache `menuBarBottom` in the poll (`:446`); template-tint the green/gray fade only after verifying light and dark bars (`isTemplate` is deliberately false at `:269`).
- **SettingsStore robustness.** `previousBundleIDs` hardcodes the base prefix (`SettingsStore.swift:42`) while `nextBundleID` derives it (`:51`), so an ID-scheme change silently loses settings across self-repair; migration is one-shot with a hardcoded key list (`:56-71`) so future keys never back-fill; migration copies untyped CFPreferences values (`:64-65`), so type-check and clamp at the boundary (cap emoji length, clamp pushMultiplier, bool-check flags). Duplicate-instance check (`AppDelegate.swift:147-155`) can race two simultaneous launches into both quitting; add a lowest-PID tiebreak.
- **install.sh and supply chain.** Add `set -euo pipefail`; kill by captured PID instead of `pkill -x` name match; add `--options runtime` to the codesign line; pin `soffes/HotKey` to an exact tag and keep `Package.resolved` committed.

## Ignore / by design

The review verified these as correct, deliberate, and hard-won. Do not "fix" them; several were explicitly proposed by individual reviewers and rejected by the synthesis.

- **Status-item creation order.** Toggle created first = rightmost = always visible (`AppDelegate.swift:8-13`). The "swap for discoverability" idea is a misread; the app is already `[hidden] │ [dot]`, and the watchdog, position keys (`mbb3-toggle`/`mbb3-separator`), and Tahoe fix all depend on this order.
- **The ExceptionShield / `safelySetLength` shield-and-fallback pattern.** Keep it; only fix the filter direction (above).
- **`itemsLookOrphaned` both-states design** (`AppDelegate.swift:776-790`): deliberately does not go blind while collapsed. Correct as-is.
- **`healLayout` repositions without killing the process** (`:697-705`) and the collapse-verification revert mechanism (`:746`): both exactly right per HANDOFF; only add the attempt guard.
- **Graceful SIGTERM/SIGINT via DispatchSource on the main queue** (`:160-169`): structurally async-signal-safe. Keep.
- **`normalizeSavedPositions` 5000 ceiling** (`:663`) gates icon positions, not `NSStatusItem.length`. "Raise it to 10000" is a category error; leave it.
- **install.sh's graceful quit dance** (osascript quit, sleep, pkill, sleep, open): never replace with a hard kill; a violent kill mid-status-item is the blacklist trigger.
- **No Accessibility permission.** Mouse polling instead of a global event monitor is deliberate; do not add the entitlement.
- **Watchdog cadences (~3s and ~5s)** catch the Tahoe orphaning live; relocating them to a dedicated tolerant timer is fine, lengthening them is not.
- **Ad-hoc signing** is a deliberate choice for a locally built tool (hardened runtime is the pragmatic floor; identity change is an open question, and commercial distribution changes this calculus entirely).
- **Rejected scope creep, unanimous:** no state-machine rewrite of collapse, no CVDisplayLink for the rainbow, no CAGradientLayer wave approximation, no migration to a SwiftUI Settings scene / Cmd+, window lifecycle (it would break the `.regular`/`.accessory` activation-policy toggle at `:895`/`:905`), no new subsystems (presets browser, floating preview window, tabbed emoji browser, hotkey recorder as a launch blocker).

## Open questions for Ga'noh

- **The dead-toggle decision (blocks the Settings rebuild).** Delete the custom-emoji-for-the-toggle feature and make the animated dot the app's fixed identity (front-end, UX, and design all lean this way), or invest in making a custom toggle emoji actually render end-to-end (which sacrifices the dot's hover/collapse/rainbow feedback)? This removes a user-facing capability the quick-pick grid exists to serve, so it needs an explicit yes.
- **Start Marker section: keep-and-fix or delete?** Fixing persistence (read `store.startEmoji`, length-safe) satisfies both camps; deletion is defensible since the README frames the boundary as a Cmd-drag concept, not an emoji people restyle.
- **Debug harnesses.** HANDOFF.md documents them as intentionally kept, but security flags them as release attack surface (`--force-repair` bypasses the confirmation; `--test-collapse` enumerates other apps' menu-bar windows). Recommendation: `#if DEBUG` plus confirmation-gating `--force-repair`. Confirm intent before removing anything.
- **Tahoe blacklist keying (gates the signing-oracle fix).** Is the macOS 26 blacklist keyed to bundle ID only, or also to on-disk path? The temp-copy-then-resign fix assumes a re-signed copy launched from a temp location won't trip it. Needs a live-fire test on this machine before shipping.
- **Signing identity, and the commercialization elephant.** Stay ad-hoc + hardened runtime, or move to the on-machine "Apple Development" identity (a dev, non-distribution cert that must be tested through the full self-repair relaunch first)? Bigger picture: selling the app implies Developer ID + notarization eventually, and both collide with a self-modifying, self-re-signing bundle. Related: should the bundle-ID-bump self-repair exist at all long-term if Apple fixes the blacklist behavior?
- **Multi-display use.** Do you (and your expected customers) run a wider secondary display? If yes, promote the `NSScreen.main` fix into the fix-before-charging batch, since collapse can be incomplete and hover can fail entirely on the secondary bar today.
- **Build/verify gate.** All findings are static and line-verified only; no reviewer built or live-tested. Before shipping any batch: branch, `swift build -c release`, `./install.sh`, then live-fire collapse/expand/rainbow/self-repair on this exact macOS 26 machine, per the hardening history.
